# BeeWorks 官网 AI 顾问系统 Roadmap

存档日期：2026-05-07

> 重要更新：本文件记录 BeeWorks 官网 AI 顾问早期方案，其中关于 Dify + Chatwoot 作为默认底座的描述已被后续架构决策更新。后续执行以 `docs/beepilot-product-roadmap-and-architecture.md` 为准：BeeWorksAI 是默认 AI Provider，自家 IM 是默认 Conversation Channel，Dify / Chatwoot 均作为可选适配器。

## 1. 系统定位

BeeWorks 官网 AI 顾问系统是 BeePilot 的第一个落地场景。BeePilot 不是普通客服浮窗，而是面向企业客户互动的 AI 售前、AI 客服和增长转化平台。

一期目标从官网场景切入，覆盖：

- 用户在官网直接咨询产品、方案、报价、试用和售后问题。
- 运营人员可以录入知识库。
- 管理人员可以查看会话记录、线索和售后请求。
- 团队可以持续优化 AI 的回答质量。

长期目标是将 BeePilot 产品化，既服务 BeeWorks 官网，也可独立销售给任意企业客户，形成从客户访问、产品理解、线索收集、试用申请、报价咨询、购买引导到售后支持的完整闭环。

## 2. 当前原型基线

当前官网方向以方案 C 亮色官网为基础，已经确定：

- 官网首页采用正式官网风格，而不是内部设计稿展示。
- 主要板块包括：Hero、产品矩阵、BeeWorks AI、行业方案、客户案例、客户覆盖、部署信创、安全合规、底部 CTA。
- AI 顾问以右侧侧边栏方式嵌入。
- 桌面端打开 AI 顾问时，官网主体向左收缩，侧边栏不覆盖主体内容。
- AI 顾问侧边栏宽度当前为 380px。
- 侧边栏内部不再嵌套二级卡片，整体作为完整 AI 顾问面板。
- 已预留百度、Google、CRM、数据仓库等分析接入的 analytics facade。

当前可访问入口：

- `official.html`：正式官网主页原型。
- `ai-plg.html`：AI PLG 官网单页入口。
- `index.html` / `index-bundle.html`：设计方案总览入口。

## 2.1 技术选型结论

当前系统底座采用：

- BeePilot Web Widget 自研：保持客户网站视觉、AI 侧边栏、快捷问题、CTA 和转化卡片的品牌体验。
- Dify：作为 AI 大脑，负责知识库、RAG、意图识别、工作流、回答生成和来源引用。
- Chatwoot：作为客服会话底座，负责会话记录、人工接管、客户资料、团队收件箱、标签和分配。
- BeePilot AI Gateway：作为自建编排层，连接网站前台、Dify、Chatwoot、CRM、工单、企业微信、邮件和分析系统。

推荐运行链路：

```text
客户网站 / BeeWorks 官网
        ↓
BeePilot Web Widget
        ↓
BeePilot AI Gateway
        ↓
  ┌──────────────┬───────────────┐
  │ Dify          │ Chatwoot       │
  │ AI / RAG / 工作流 │ 会话 / 人工接管 / 客户资料 │
  └──────────────┴───────────────┘
        ↓
CRM / 工单 / 企业微信 / 邮件 / 数据分析
```

选型原则：

- 不直接使用 Chatwoot 默认网站 widget，避免破坏客户网站设计。
- 前台不直接调用 Dify 或 Chatwoot，避免 API Key 暴露。
- AI Gateway 统一处理会话 ID、用户归因、业务动作、风控边界和第三方系统同步。
- 后台重点表达 BeePilot 运营控制台，真实会话和人工接管进入 Chatwoot。
- 知识库和工作流后续优先在 Dify 配置，BeeWorks 后台提供入口、状态和业务化封装。

## 4. 知识库设计

### 4.1 知识库分类

一期建议按 12 类准备资料：

1. 公司与品牌
2. 产品总览
3. 功能模块
4. AI 能力
5. 行业方案
6. 客户案例
7. 部署与信创
8. 安全与合规
9. 价格与商务
10. 实施交付
11. 售后支持
12. 禁止承诺与风险边界

### 4.2 知识条目字段

每条知识建议包含：

- 标题
- 分类
- 标签
- 适用行业
- 适用模块
- 适用版本
- 内容正文
- 可见范围：官网可答 / 内部参考 / 禁止外发
- 状态：草稿 / 待审核 / 已发布 / 已下线
- 更新时间
- 负责人

### 4.3 资料形态

一期原型支持：

- 手工 FAQ
- 文本片段
- 文档条目
- 链接条目

后续再扩展：

- PDF / Word / PPT / Excel 上传解析
- 网页抓取
- 向量化检索
- 知识版本对比
- 知识有效期提醒

## 5. AI 工作流设计

一期内置 5 条核心工作流。

### 5.1 产品咨询

触发：

- 用户询问 BeeWorks 是什么、有哪些功能、和其他系统有什么区别。

流程：

1. 识别关注点。
2. 检索产品总览和功能模块知识。
3. 回答产品能力。
4. 推荐相关模块或案例。
5. 引导预约演示、试用或继续追问。

### 5.2 行业方案

触发：

- 用户提到金融、央国企、制造、高校、医疗、政务等行业。

流程：

1. 识别行业。
2. 追问规模、部署方式、核心诉求。
3. 检索行业方案和客户案例。
4. 输出方案摘要。
5. 引导下载方案、留资或预约顾问。

### 5.3 报价咨询

触发：

- 用户问价格、报价、收费、授权、500 人多少钱等。

流程：

1. 判断为高风险意图。
2. 收集人数、模块、部署方式、是否信创、是否需要 AI。
3. 检索价格与商务知识。
4. 只给报价影响因素或参考口径。
5. 正式报价转人工销售。

边界：

- AI 不承诺最终价格。
- AI 不承诺折扣。
- AI 不承诺合同条款。

### 5.4 试用申请

触发：

- 用户要求试用、演示、POC。

流程：

1. 收集姓名、公司、联系方式、行业、人数。
2. 判断在线试用、演示环境或私有化 POC。
3. 创建试用申请或演示预约。
4. 记录线索。
5. 后续同步 CRM 或通知销售。

### 5.5 售后支持

触发：

- 用户提到登录、账号、会议、文档、故障、使用问题、已有客户等。

流程：

1. 判断是否已有客户。
2. 收集公司、问题类型、影响范围、紧急程度。
3. 检索售后知识库。
4. 可以解决则给步骤。
5. 不能解决则创建售后请求或转人工。

## 6. 配置中心设计

后台需要让运营人员配置 AI，而不是把规则写死在代码里。

### 6.1 工作流配置字段

每条工作流包含：

- 工作流名称
- 触发意图
- 触发关键词
- 需要收集的字段
- 可调用知识库分类
- 允许动作
- 禁止动作
- 转人工条件
- 回答模板
- 风险等级
- 是否启用

示例：

```json
{
  "id": "quote_request",
  "name": "报价咨询",
  "trigger_intents": ["询价", "报价", "价格", "收费"],
  "required_fields": ["company", "contact", "user_count", "deployment_type", "modules"],
  "knowledge_scopes": ["价格与商务", "产品总览", "部署与信创"],
  "allowed_actions": ["ask_followup", "create_lead", "handoff_sales"],
  "forbidden_actions": ["commit_final_price", "commit_discount"],
  "risk_level": "high",
  "success_state": "创建报价线索"
}
```

### 6.2 表单字段配置

可复用字段：

- 姓名
- 公司
- 手机
- 邮箱
- 行业
- 组织人数
- 部署方式
- 关注模块
- 当前系统
- 预算阶段
- 问题紧急程度

### 6.3 动作配置

AI 可触发动作：

- 创建销售线索
- 创建报价申请
- 创建试用申请
- 创建售后请求
- 转人工
- 推荐资料下载
- 推送企业微信
- 同步 CRM
- 发送邮件

一期原型先模拟这些动作并保存记录。

## 7. 会话记录与优化闭环

所有 AI 对话必须留痕。

### 7.1 会话记录字段

- 会话 ID
- 用户问题
- AI 回答
- 会话时间
- 来源页面
- 来源渠道：utm、gclid、bd_vid、referrer
- 识别意图
- 命中知识库
- 是否留资
- 是否创建线索
- 是否创建售后请求
- 是否转人工
- AI 置信度
- 人工评价

### 7.2 回答优化

后台支持：

- 标记回答：好 / 一般 / 不准确 / 不应回答
- 编辑推荐回答
- 保存为 FAQ
- 标记知识缺口
- 下线错误知识
- 查看高频未解决问题

### 7.3 知识缺口机制

系统自动汇总：

- AI 没答上的问题
- 用户重复追问的问题
- 被人工标记不准确的问题
- 高频但无知识库命中的问题

这些问题进入“待补充知识”列表。

## 9. 分阶段 Roadmap

### 阶段 0：当前官网原型收敛

状态：进行中

目标：

- 方案 C 正式官网主页成型。
- AI 顾问侧边栏完成基础交互。
- 官网打开 AI 顾问时自适应收缩。
- BeeWorks 图标替换完成。
- 蓝白区块节奏统一。

交付：

- `official.html`
- `src/site-ai-plg.jsx`
- `src/analytics.jsx`

### 阶段 1：后台原型

目标：

- 做出 Chatwoot + Dify 架构下的 AI 顾问运营后台静态原型。

交付：

- `admin.html`
- `src/admin-ai-advisor.jsx`
- 后台左侧导航
- Dify 知识库管理界面
- Dify 工作流配置界面
- Chatwoot 会话与人工接管界面
- 线索与售后请求界面
- 回答优化界面
- Gateway 配置界面

验收标准：

- 能看清系统怎么运营。
- 能模拟新增知识库条目。
- 能模拟配置工作流。
- 能查看 mock 会话、人工接管状态和线索。
- 能标记回答质量。
- 能看清 Dify、Chatwoot、Gateway 的职责边界。

### 阶段 2：前台与后台数据打通 mock

目标：

- 让官网 AI 顾问和后台记录形成闭环，即使数据仍是本地 mock。

交付：

- 前台提问后生成会话记录。
- 快捷问题触发对应意图。
- 留资、报价、试用、售后动作进入后台列表。
- 回答反馈进入优化模块。

验收标准：

- 能演示“用户提问 -> AI 回答 -> 留资/售后 -> 后台可查看”。

### 阶段 3：知识库检索原型

目标：

- 从静态回答升级为基于知识条目的检索回答。

交付：

- 知识库条目检索。
- 按分类和标签匹配。
- 回答显示参考来源。
- 无命中时引导转人工或补充信息。

验收标准：

- AI 回答可追溯到知识库来源。
- 不在知识库里的问题不会编造答案。

### 阶段 4：真实后端 MVP

目标：

- 从静态原型进入可保存、可多人运营的系统。

交付：

- 用户、知识库、工作流、会话、线索、售后请求数据表。
- 管理后台登录。
- 文件上传和文本解析。
- API：
  - `POST /api/chat`
  - `POST /api/knowledge`
  - `POST /api/leads`
  - `POST /api/quotes`
  - `POST /api/trials`
  - `POST /api/support`
  - `POST /api/events`

验收标准：

- 数据真实保存。
- 后台可编辑发布知识。
- 官网 AI 可读取已发布知识。

### 阶段 5：接入真实 AI 与 RAG

目标：

- 接入大模型、向量检索和工作流编排。

交付：

- 文档切分和向量化。
- 知识库检索。
- 意图识别。
- 工作流状态机。
- 回答引用来源。
- 低置信度转人工。

验收标准：

- AI 能基于知识库回答官网访客问题。
- 回答边界可控。
- 每次回答有日志和来源。

### 阶段 6：业务系统集成

目标：

- 将 AI 顾问接入真实业务流程。

交付：

- CRM 线索同步。
- 企业微信/邮件通知。
- 试用申请流转。
- 报价申请流转。
- 工单系统对接。
- GA4、百度统计、广告转化追踪接入。

验收标准：

- AI 产生的有效线索可进入销售流程。
- 售后问题可进入工单流程。
- 渠道到线索可归因。

### 阶段 7：运营优化

目标：

- 用数据持续提升 AI 顾问转化率和解决率。

交付：

- 高频问题分析。
- 知识缺口分析。
- 回答质量分析。
- 渠道转化漏斗。
- 工作流 A/B 测试。
- 销售跟进效果回流。

核心指标：

- AI 打开率
- 快捷问题点击率
- 问题解决率
- 知识库命中率
- 留资转化率
- 试用申请率
- 报价申请率
- 售后自助解决率
- 人工转接率
- 有效线索率
