# BeeWorks 官网 + AI PLG 闭环方案

## 1. 项目目标

BeeWorks 官网不只承担品牌展示和线索收集，而是升级为一个由 AI 驱动的 PLG 转化入口。用户进入官网后，可以通过自然语言完成产品了解、行业方案匹配、留资、试用、报价、购买咨询和售后问题解答，形成从访问到转化、从售前到售后的完整闭环。

核心目标：

- 提升官网首屏转化效率。
- 降低用户理解企业协同、私有化部署、信创适配等复杂能力的成本。
- 用 AI 顾问承接售前、试用、报价、采购和售后意图。
- 通过行为采集和归因体系衡量每条转化路径的有效性。
- 为后续接入 CRM、试用系统、报价系统、支付系统、工单系统和知识库预留架构。

## 2. 视觉与首页方向

推荐以当前原型中的 **A 科技流** 作为官网主页基础。

原因：

- BeeWorks 的核心卖点是企业协同、私有化部署、AI、信创和数据主权，A 版的技术感和平台感更符合大客户采购心智。
- A 版产品界面 mock、数据卡片、AI 执行链和信创矩阵更容易建立可信度。
- 相比 C 亮色流，A 版更适合表达安全、稳定、私有化和企业级能力。
- 相比 B Editorial 流，A 版转化效率更强，信息密度和产品感更适合作为官网首页。

建议视觉组合：

- A 科技流占 70%：作为首页主基调。
- B Editorial 占 20%：用于案例、关于、信创、安全合规等权威内容区。
- C 亮色流占 10%：用于表单、报价、试用、购买、售后等降低填写压力的交互卡片。

## 3. 首页信息架构

首页保留传统官网的可信内容，同时把 AI 顾问提升为首屏核心交互。

推荐结构：

1. 首屏 Hero
   - 左侧：品牌主张、价值说明、核心 CTA。
   - 右侧：AI 顾问面板，不再只是静态产品 mock。
   - 主标题建议：`私有化部署的企业协同与 AI 办公平台`
   - 副标题建议：`IM、会议、文档、流程、知识库、AI 助手一体化，数据留在企业自己的环境里。`
   - CTA：`问 AI 顾问`、`开始试用`、`获取报价`。

2. 产品矩阵
   - 即时通讯、视频会议、文档协同、流程引擎、日程会议、企业网盘、人事中台、知识库、AI 助手、低代码、电子签章、数据 BI。

3. AI 能力
   - 智能纪要、知识问答、流程发起、Agent 编排、本地大模型部署。

4. 行业方案
   - 金融、央国企、制造、高校、医疗、政务。

5. 客户案例
   - 按行业展示关键指标、部署方式、效果数据。

6. 私有化部署与信创适配
   - CPU、操作系统、中间件、数据库、应用层适配矩阵。

7. 安全合规
   - 数据零出域、等保三级、商密评估、高可用架构、离线可用。

8. 新闻动态与资料下载
   - 白皮书、行业方案、信创适配矩阵、产品手册。

9. 底部 CTA
   - 再次引导问 AI、预约演示、获取报价、开始试用。

## 4. AI 顾问设计

AI 顾问不是悬浮客服插件，而是官网首屏核心模块。

首屏快捷问题：

- 我想了解私有化部署
- 帮我估算 500 人团队价格
- 我想申请试用
- 我们是制造业，有没有方案
- 我已经是客户，需要售后支持
- 帮我生成一份选型建议

AI 对话中的动作卡片：

- 留资卡片
- 预约演示卡片
- 试用创建卡片
- 报价估算卡片
- 行业方案推荐卡片
- 售后工单卡片
- 资料下载卡片
- 转人工卡片

页面联动能力：

- 用户问行业方案时，页面自动滚动并高亮对应行业模块。
- 用户问客户案例时，推荐相关案例卡片。
- 用户问信创和安全时，跳转到信创适配矩阵或合规区。
- 用户问价格时，打开报价估算卡片。
- 用户问试用时，打开试用创建卡片。

## 5. PLG 闭环路径

第一版重点设计 5 条路径。

### 5.1 产品了解

路径：

用户提问 -> AI 解释能力 -> 推荐产品模块 -> 推荐案例或资料 -> 引导预约演示或试用。

关键动作：

- 产品模块曝光
- CTA 点击
- 资料下载
- 预约演示

### 5.2 行业方案

路径：

用户输入行业、规模、部署要求 -> AI 推荐行业方案 -> 展示案例和指标 -> 引导留资或下载方案。

关键动作：

- 行业识别
- 案例点击
- 行业资料下载
- 线索提交

### 5.3 报价咨询

路径：

用户输入人数、模块、部署方式、是否信创 -> AI 给价格区间和影响因素 -> 复杂私有化场景转销售。

关键动作：

- 报价估算开始
- 报价参数填写
- 报价请求提交
- 销售线索创建

边界：

- AI 只给参考区间。
- 正式报价必须进入销售流程。
- 不承诺固定交付周期和合同价格。

### 5.4 自助试用

路径：

用户申请试用 -> 填写公司、联系人、手机号、邮箱、行业、规模 -> 创建试用空间或提交开通申请 -> 邮件/短信通知。

关键动作：

- 试用入口点击
- 试用表单开始
- 试用提交成功
- 试用空间创建成功

### 5.5 售后支持

路径：

用户描述问题 -> AI 查询知识库 -> 返回解决步骤 -> 无法解决则创建工单或转人工。

关键动作：

- 售后意图识别
- 知识库命中
- 工单创建
- 人工转接

边界：

- 涉及账号、合同、发票、故障数据时需要身份验证。
- AI 不直接暴露客户敏感信息。

## 6. 增长分析与行为采集

官网需要从第一版开始预留统一的分析和埋点层，避免后续把百度、Google、广告追踪代码散落在业务组件里。

### 6.1 需要支持的第三方工具

预留接入：

- 百度统计
- Google Analytics / GA4
- Google Tag Manager
- 百度营销转化追踪
- Google Ads 转化追踪
- 巨量引擎、腾讯广告、知乎等广告平台
- CRM / SCRM
- 自建数据仓库或 BI

### 6.2 统一 analytics 层

建议提供统一调用接口：

```js
analytics.track("lead_submit", {
  source: "ai_chat",
  intent: "quote",
  company_size: "500",
  industry: "manufacturing"
});
```

业务组件只调用统一事件接口，不直接调用 `gtag()`、百度 `_hmt` 或其他厂商 SDK。

### 6.3 推荐配置结构

```js
window.BEEWORKS_ANALYTICS = {
  ga4: {
    enabled: true,
    measurementId: "G-XXXX"
  },
  gtm: {
    enabled: false,
    containerId: "GTM-XXXX"
  },
  baidu: {
    enabled: true,
    siteId: "XXXX"
  },
  consentRequired: true
};
```

### 6.4 第一版关键事件

页面行为：

- `page_view`
- `scroll_depth`
- `nav_click`
- `hero_cta_click`
- `product_module_view`
- `product_module_click`
- `industry_tab_click`
- `case_click`
- `download_click`

AI 行为：

- `ai_chat_open`
- `ai_quick_prompt_click`
- `ai_message_send`
- `ai_intent_detected`
- `ai_recommendation_click`
- `ai_action_card_show`
- `ai_action_card_submit`
- `ai_handoff_requested`

转化行为：

- `lead_form_open`
- `lead_form_start`
- `lead_submit`
- `demo_book`
- `trial_start`
- `trial_submit`
- `quote_request`
- `purchase_start`
- `support_request`
- `ticket_create`

### 6.5 归因参数

需要在访问时采集并在后续转化动作中携带：

- `utm_source`
- `utm_medium`
- `utm_campaign`
- `utm_content`
- `utm_term`
- `gclid`
- `bd_vid`
- `referrer`
- `landing_page`
- `first_touch`
- `last_touch`

这些参数需要随留资、试用、报价、购买、工单一起进入后端和 CRM。

### 6.6 合规与隐私

预留 Cookie / 隐私同意机制：

- 首次访问展示隐私提示。
- 用户同意后加载营销追踪脚本。
- 基础匿名统计和营销追踪分层控制。
- 表单提交时明确说明数据用途。
- AI 对话内容需要敏感信息提示和必要脱敏。

## 8. AI 回答边界

必须提前定义 AI 的可回答范围。

规则：

- 报价只能给参考区间，正式报价进入销售流程。
- 合规、安全、信创问题必须引用资料来源。
- 私有化部署周期只能给参考范围。
- 售后问题需要判断是否已有客户。
- 涉及合同、发票、账号、故障数据时转人工或工单。
- 所有留资和试用动作都需要用户明确确认。
- AI 不编造客户案例、认证、价格、交付承诺。

知识库建议：

- 官网公开内容
- 产品手册
- 私有化部署白皮书
- 信创适配矩阵
- 安全合规材料
- 行业方案
- 客户案例
- FAQ
- 售后知识库

## 9. 分阶段执行计划

### 阶段 1：首页原型重构

目标：把当前 A 版首页改造成 AI First 官网首页。

交付：

- 新首屏 Hero。
- AI 顾问面板静态交互。
- 三个核心 CTA。
- 保留产品矩阵、AI、行业、案例、部署、合规内容。

### 阶段 2：AI 对话与动作卡片原型

目标：完成可点击的 AI PLG 流程。

交付：

- 6 个快捷问题。
- 5 条主路径对话状态。
- 留资、试用、报价、工单、资料下载卡片。

### 阶段 3：埋点与分析基础设施

目标：完成统一 analytics 层和关键事件模型。

交付：

- analytics loader。
- 事件命名规范。
- UTM / gclid / bd_vid 采集。
- 百度统计、GA4、GTM 配置占位。
- 首批关键事件接入。

### 阶段 4：假 API 跑通闭环

目标：在不接真实系统的情况下跑通全流程。

交付：

- `/api/chat` mock。
- `/api/leads` mock。
- `/api/trials` mock。
- `/api/quote` mock。
- `/api/tickets` mock。
- `/api/events` mock。

### 阶段 5：接入真实业务系统

目标：从原型进入可上线版本。

交付：

- CRM 线索入库。
- 试用空间创建。
- 报价请求流转。
- 售后工单创建。
- 资料下载记录。
- 行为数据进入分析系统。

### 阶段 6：上线灰度与优化

目标：验证 AI PLG 官网是否提升转化。

上线顺序：

1. 售前咨询、留资、预约演示。
2. 自助试用。
3. 报价和购买咨询。
4. 售后工单和客户知识库。

核心指标：

- AI 面板打开率
- 快捷问题点击率
- 留资转化率
- 试用创建率
- 报价咨询完成率
- 预约演示率
- 售后自助解决率
- 人工转接率
- 渠道到有效线索转化率

## 10. 当前项目中的下一步建议

基于现有目录，建议下一步直接从原型层推进：

1. 在当前 `src/site-tech.jsx` 基础上新建或改造 AI First 首页首屏。
2. 在 `src/shared.jsx` 中补充 AI 快捷问题、意图、动作卡片 mock 数据。
3. 新增前端 analytics 工具文件，先做 no-op + console 模式。
4. 在关键 CTA 和 AI 交互上接入事件调用。
5. 保留当前 A/B/C 画板对比，但新增一个最终方向画板：`D · AI PLG 官网主页`。
